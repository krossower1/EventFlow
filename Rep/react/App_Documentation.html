1. Przegląd Komponentu

Nazwa: App (Default Export)

Typ: React Functional Component

Przeznaczenie: Główny komponenty aplikacji EventFlow - system zarządzania wydarzeniami z obsługą autentykacji, różnych ról użytkownika i wielopoziomowego interfejsu.

Importy: React (hooks), axios (HTTP), CSS styles

Główne Odpowiedzialności:

  • Zarządzanie stanem autentykacji (login/rejestracja)
  • Wyświetlanie interfejsu na podstawie roli użytkownika (USER, ORG, ADMIN)
  • Obsługa formularzy dla miejsc, sal i wydarzeń
  • Komunikacja z API backend (http://localhost:8081/api)
  • Nawigacja między zakładkami i sekcjami aplikacji
  • Zarządzanie wniioskami o rolę organizatora (dla ADMIN)

3. Funkcje Pomocnicze

Autentykacja

getAuthHeaders()

Zwraca: object { Authorization: Basic <base64> }

Opis: Tworzy nagłówki HTTP Basic Auth z aktualnych poświadczeń zalogowanego użytkownika. Użytkiwane do wszystkich żądań API.

const basicToken = btoa(`${authCredentials.login}:${authCredentials.password}`); return { Authorization: `Basic ${basicToken}` };

Wyszukiwanie i Filtrowanie

filteredWydarzenia (computed)

Zwraca: array

Opis: Filtruje listę wydarzeń na podstawie tekstu wyszukiwania i filtera statusu.

Logika filtracji:

  • Szuka w polu: tytul, miejsceNazwa (bez rozróżniania wielkości)
  • Filtruje po statusie: 'ALL' (wszystkie) lub konkretny status (AKTYWNE, SZKIC, ZAMKNIETE)

4. Obsługiwacze Formularzy (Form Handlers)

onLoginSubmit(event)

Parametr: event (FormEvent)

Akcje:

  • POST do: http://localhost:8081/api/auth/login
  • Wysyła: loginForm (login, password)
  • Przy sukcesie: ustawia logowanie, poświadczenia, rolę
  • Obsługuje błędy z komunikatami statusu

onRegisterSubmit(event)

Parametr: event (FormEvent)

Akcje:

  • POST do: http://localhost:8081/api/auth/register
  • Wysyła: registerForm (imie, nazwisko, email, login, password)
  • Przy sukcesie: przełącza na tryb logowania
  • Obsługuje błędy z komunikatami

onMiejsceSubmit(event)

Parametr: event (FormEvent)

Akcje:

  • POST do: http://localhost:8081/api/miejsca
  • Wysyła: miejsceForm + konwersja pojemnosci na Number
  • Przy sukcesie: resetuje formularz, odświeża listę
  • Wymaga: Basic Auth headers

Wymagane pola: nazwa, miasto, ulica, kodPoczt, pojemnosc

onSalaSubmit(event, miejsceId)

Parametry: event (FormEvent), miejsceId (number)

Akcje:

  • POST do: http://localhost:8081/api/miejsca/{miejsceId}/sale
  • Wysyła: { nazwa, pojemnosc, pietro, maPlan }
  • Konwertuje pojemnosc i pietro na Number
  • Resetuje formularz dla tego miejsca po sukcesie

onWydarzenieSubmit(event)

Parametr: event (FormEvent)

Akcje:

  • POST do: http://localhost:8081/api/wydarzenia
  • Wysyła: wszystkie pola z wydarzenieForm
  • Obsługuje tworzenie nowych kategorii na lotu
  • Odświeża opcje i listę wydarzeń po sukcesie

Logika kategorii: Jeśli createNowaKategoria = true, wysyła nowaKategoriaNazwa i nowaKategoriaOpis

onOrganizerRequestSubmit(event)

Parametr: event (FormEvent)

Akcje:

  • POST do: http://localhost:8081/api/organizator/request
  • Wysyła: organizerForm (firma, kwalifikacje, strona)
  • Resetuje formularz po sukcesie
  • Wymaga: Basic Auth

updateSalaForm(miejsceId, field, value)

Parametry: miejsceId (number), field (string), value (any)

Opis: Aktualizuje pole w salaForms dla konkretnego miejsca. Zachowuje pozostałe pola.

5. Funkcje Pobierania Danych

fetchUsers()

Endpoint: GET http://localhost:8081/api/users

Ustawia: data (array użytkowników), loading (boolean)

Wymaga: Basic Auth headers

Wywoływana: useEffect gdy isLoggedIn = true

fetchOrganizerRequests()

Endpoint: GET http://localhost:8081/api/organizator

Ustawia: organizerRequests (array), organizerLoading (boolean)

Wymaga: Basic Auth headers

Used by: ADMIN role

fetchMyMiejsca()

Endpoint: GET http://localhost:8081/api/miejsca/my

Ustawia: miejsca (array), miejscaLoading (boolean)

Wymaga: Basic Auth headers

Used by: ORG role

fetchWydarzeniaOptions()

Endpoint: GET http://localhost:8081/api/wydarzenia/options

Ustawia: wydarzenieOptions (array { miejsca, kategorie }), wydarzenieLoading (boolean)

Wymaga: Basic Auth headers

Użycie: Pobiera opcje dla formularza nowego wydarzenia

fetchMyWydarzenia()

Endpoint: GET http://localhost:8081/api/wydarzenia/my

Ustawia: myWydarzenia (array)

Wymaga: Basic Auth headers

Used by: ORG role

ℹ️ Efekty uboczne (useEffect):
  • fetchUsers() jest wywoływana gdy isLoggedIn lub authCredentials się zmienią
  • Dla roli ORG: fetchMyMiejsca(), fetchWydarzeniaOptions(), fetchMyWydarzenia()

Akcje dla Roli ADMIN

onApproveOrganizer(id)

Endpoint: POST http://localhost:8081/api/organizator/{id}/approve

Akcje:

  • Zatwierdza wniosek organizatora
  • Odświeża listę wniiosków i użytkowników
  • Wysyła komunikat sukcesu/błędu

onRejectOrganizer(id)

Endpoint: DELETE http://localhost:8081/api/organizator/{id}/reject

Akcje:

  • Odrzuca i usuwa wniosek organizatora
  • Odświeża listę wniiosków
  • Wysyła komunikat statusu

Kluczowe Cechy i Przepływy

1. Przepływ Logowania

  • Użytkownik wpisuje login i hasło
  • System wysyła POST do /api/auth/login
  • Backend zwraca response z polem success i rola
  • Ustawiana jest zmienna isLoggedIn = true
  • Pobierane są dane dla odpowiedniej roli

2. Przepływ Wniosku Organizatora

  • USER klika "Zostań Organizatorem"
  • Wypełnia formularz (firma, kwalifikacje, strona)
  • System wysyła POST do /api/organizator/request
  • ADMIN otrzymuje wniosek w sekcji "Wnioski organizatora"
  • ADMIN zatwierdza (approve) → rola zmieniana na ORG
  • ADMIN odrzuca (reject) → wniosek usuwany

3. Przepływ Tworzenia Wydarzenia (ORG)

  • ORG kliknie "+ Nowe wydarzenie"
  • Pobierane są opcje (miejsca, kategorie) za helper fetchWydarzeniaOptions()
  • Wypełnia formularz z:
    • Wyborem miejsca ze selectu
    • Nową kategorią (jeśli oznaczy checkbox)
    • Datami rozpoczęcia i zakończenia
  • System wysyła POST do /api/wydarzenia
  • Wydarzenia pojawia się na liście

4. Filtrowanie Wydarzeń

  • ORG może wyszukiwać po tekście (tytuł, miejsca)
  • Może filtrować po statusie: ALL, AKTYWNE, SZKIC, ZAMKNIETE
  • FilteredWydarzenia obliczane jest dynamicznie

5. Wylogowanie

  • Klik na menu konta → "Wyloguj"
  • Resetowanie całego state'u do wartości początkowych
  • Powrót do ekranu logowania

Potencjalne Ulepszenia

⚠️ Uwagi o Kodzie:

  • Duży Component: Komponent App.js jest bardzo duży (~1000+ linii). Rozważyć rozdzielenie na mniejsze komponenty.
  • Bezpieczeństwo: Hasła i poświadczenia przechowywane w state. Rozważyć przechowywanie w secure cookies.
  • Brak ValidaUsed: Brak walidacji danych na froncie (email validation, format sprawdzenie).
  • Error Handling: Błędy obsługiwane głównie poprzez status wiadomości bez globalnego error boundary.
  • Placeholder Zawartość: Wiele sekcji to placeholder (Bilety, Miejsca, Analityka, Ustawienia).
  • Brak Cache'u: Każde przełączenie tab pobiera dane na nowo.

✅ Sugestie Refaktoryzacji:

  • Wydzielić logikę autentykacji do osobnego hooeka useAuth()
  • Stworzyć komponenty dla każdej roli (UserPanel, AuthPanel, AdminPanel)
  • Użyć Context API lub Redux do zarządzania globalnym state'em
  • Wydzielić API calls do osobnego serwisu
  • Dodać React Query lub useMemo dla cachingu danych
  • Tworzyć custom hooki dla operacji (useFormHandler, useFetchData)
